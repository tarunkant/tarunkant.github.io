    Unusual Cache Poisoning between Akamai and S3 buckets

    SpyClub
    2022-12-14
    blog
    AWS, Akamai, Bug, Cache Poisoning, S3

    Hey, I am SpyD3r(@TarunkantG) and in this blog, I will be presenting an unusual way of Cache Poisoning which happens between Akamai and Amazon S3 Buckets.

    What is Cache Poisoning?

    Web cache poisoning is a technique whereby an attacker exploits the behaviour of a web server and caches the harmful HTTP response so that it can be served to other users.
    A poisoned web cache can lead to numerous different types of web attacks such as XSS, JavaScript Injection, Open Redirection, DoS, etc.

    Learn about Cache Poisoning in detail from the PortSwigger Blog.

    What is the Issue here?

    Akamai was observed to load pages from a different origin (i.e., malicious host) than what is specified in the Akamai configuration.
    For example, let’s say redacted.example.com is mapped to redacted.s3.ap-south-1.amazonaws.com in Akamai; now using this issue an attacker can force redacted.example.com to serve a response of malicious host such as tarunkant.s3.ap-south-1.amazonaws.com. And if caching is enabled for such hosts then the malicious response will be cached and it can be served to victim users for exploitation purposes

    Exploitation

    This issue can be exploited by injecting malicious Host headers that contain some specific separators’ Unicode characters. One character that I used in this technique is \x1d.

    Notice the header \x1dhost which contains a malicious bucket ap-south-1.tarunkant.com that is created by the attacker and hosts an evilsvgfile.svg file that contains malicious XSS payload.
    As you see the response of the cURL request, notice that the content has been fetched from the malicious bucket rather than the s3 bucket where the host was mapped to.

    There is a catch that the malicious bucket that the attacker is passing via \x1dhost header should be in the same region as of the S3 bucket which is mapped to the vulnerable host. So, How it can be done?
    We can create an S3 bucket in all regions and upload a malicious evilsvgfile.svg file in all the buckets. And after that bruteforce the value of \x1dhost header with the created bucket names on the vulnerable target to check if any part of the malicious payload came in the response.

    Why SVG File?

    While conducting the research it was noticed that .svg extension is very common and almost every time to be present in Caching configuration of Akamai. Thus it makes it easy to exploit this issue via .svg extension.

    Any requirements for the attacker’s s3 bucket?

    Yes, the attacker has to create a public bucket. It was also observed during the research that there are some edge cases where we need to enable static website hosting on the S3 bucket and hence the .com in the name of the S3 bucket. So it is also recommended to have static website hosting enabled in the S3 bucket to include testing edge cases as well.

    As this issue only affects the target which are S3 bucket and onboarded in Akamai, we need to figure out three main things on the hosts that are pointed below in order to exploit this issue at scale,:

    • Are those S3 Websites?
    • Are they Onboarded in Akamai?
    • If satisfy above both points then perform the attack

    Root Cause Analysis

    I did some analysis on why Akamai is fetching the data from another origin than what is specified in its configuration and found out that Akamai prioritizes the host header with unicode value over the original host header while making the outbound requests(i.e. Request sent from Akamai Edge servers towards customer Origin). I strongly believe that this might be happening due to incorrect parsing when inclusion of unicode characters in the header.

    As you could notice when someone makes a request with a malicious host header then that host header gets prioritized in the outbound request of Akamai. And if any services like Amazon S3 which consider the first host header from the request as valid then it will return the object from the malicious host rather than the real host.

    Also to note that Amazon S3 just fetches the first occurrence of the host header without caring if it contains any unicode character or not and performs the action on that host header.

    And if caching is enabled for the host then it can be tricked to cache malicious responses instead of the actual responses.

    Responsible Disclosure

    I have responsibly reported this security bug to the Akamai Security team, and they have acknowledged that this is a valid security issue.

    Reply from Akamai

    This undesired interaction between Akamai proxies and AWS S3 origins is a configuration issue that’s joint responsibility between our customers, AWS, and Akamai. It results from certain customers’ needs to process non-RFC compliant requests for their applications to function correctly, and therefore cloud vendors offering features to support such traffic unless explicitly instructed not to do so.

    We thank you again for your notification, we are working with the owners of the application to identify the best mitigation for the circumstances. But a blanket “patch” to correct all such instances is not trivial, as that may have a big negative impact on legitimate Internet traffic.

    PoC Video

    Who all are affected?

    All those S3 bucket domains that are onboarded in Akamai and have caching enabled might be affected from this.

    How to prevent?

    Do not accept any malicious header (i.e. header containing unicode characters) at the Akamai level itself. Additionally, Akamai has custom tags to stop accepting requests with malicious headers but they are not set by default.
